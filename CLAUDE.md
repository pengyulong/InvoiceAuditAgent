# 智能合同发票审计系统开发任务说明

## 1. 任务目标

基于现有的**产品需求文档 (PRD)**、**UI/UX 原型图**和**后端接口文档 (RD)**，高质量完成 Claude 项目的开发工作。确保最终产品的功能、界面和数据交互与文档描述**完全一致**。

## 2. 关键开发要求

### 2.1 严格遵循文档

* **产品需求 (PRD):**
    * **所有功能点**必须严格按照 PRD 中的描述实现，不得遗漏、随意增减或更改核心逻辑。
    * 特殊的用户故事（User Stories）和验收标准（Acceptance Criteria）必须全部满足。
    * 本项目的prd文档见./prd.md
* **用户界面 (UI/UX 原型图):**
    * 页面的**布局、颜色、字体、间距、交互动画**等所有视觉和交互元素，必须与原型图（例如 Figma, Sketch 等）一致。
    * 确保在不同屏幕尺寸和设备上（如适用）的响应式设计符合预期。
    * 原型图参考./interaction_flow.png
* **接口文档 (RD):**
    * 所有**数据请求 (Request)** 和 **数据响应 (Response)** 的结构、字段名称、数据类型、状态码等，必须严格按照后端接口文档定义进行。
    * **错误处理机制**需与接口文档中定义的错误码和消息体保持一致。
    * 接口文档见./rd.md

### 2.2 代码质量与架构

* **可维护性 (Maintainability):**
    * 采用清晰、一致的命名约定（变量、函数、类）。
    * 编写简洁、模块化、高内聚低耦合的代码。
    * 关键业务逻辑和复杂部分必须添加清晰的**注释**。
* **可扩展性 (Extensibility):**
    * 设计良好的**代码结构和目录组织**，方便未来新功能或模块的添加。
    * **核心业务逻辑**应与 UI 渲染和数据处理逻辑分离（例如，遵循 MVC, MVVM, 洁净架构等设计模式）。
    * 组件化/模块化：确保 UI 组件和功能模块可复用。
* **性能优化:**
    * 关注代码执行效率，避免不必要的渲染或计算。

## 3. 测试与质量保证

### 3.1 自测与联调

* 开发人员在提交流代码之前，必须进行充分的**单元测试**和**集成测试 (联调)**，确保所有接口调用和前端逻辑的正确性。

### 3.2 严格的正式测试流程

完成开发工作后，必须进入以下严格的测试阶段：

1.  **用户验收测试 (UAT - User Acceptance Testing):**
    * 由产品经理和/或业务代表根据 **PRD**，对所有功能点进行模拟真实用户的测试。
    * 目的是确认系统功能是否满足最初设定的业务需求。
2.  **系统集成测试 (SIT - System Integration Testing):**
    * 测试系统与其他相关系统（如后端服务、数据库、外部 API 等）的**集成**是否正确无误。
    * 重点关注数据流转、接口调用和异常处理。

### 3.3 Bug 记录与修复

* 在 UAT 和 SIT 过程中发现的所有 Bug，必须**详细记录**在指定的缺陷管理系统（如 Jira, Bugzilla 等）中。
* 记录内容包括：**Bug 编号、描述、重现步骤、严重程度、截图/视频（如适用）**。
* 开发人员必须**及时、准确地修复**所有已确认的 Bug，并与测试人员协作，进行回归测试，确保 Bug 已被消除且没有引入新的问题。

## 4. 交付标准

* 所有 PRD、原型图和接口文档中的要求均已实现。
* 代码已通过 Code Review，满足可维护性和可扩展性要求。
* 所有功能已通过 UAT 和 SIT 测试。
* 所有已发现的 P1/P2 级别 Bug 已修复并通过回归测试。
